<feature>
  <name>Log Buffer Ring Buffer</name>
  <files>internal/scheduler/logbuffer.go, internal/scheduler/logbuffer_test.go</files>
  <behavior>
    The logBuffer is a bounded ring buffer of LogEntry values with its own sync.Mutex (independent of any Scheduler mutex — this is a locked architectural decision from STATE.md to prevent Phase 6 deadlocks).

    Types to define:
    - Stream type (string): StreamStdout = "stdout", StreamStderr = "stderr"
    - LogEntry struct: Timestamp (time.Time), Stream (Stream), Text (string)
    - logBuffer struct (unexported): mu (sync.Mutex), entries ([]LogEntry), size (int), head (int), count (int)

    Constructor:
    - newLogBuffer(size int) *logBuffer — if size <= 0, default to 1000. Allocates entries slice of given size.

    Methods (all pointer receiver on *logBuffer):
    - Write(entry LogEntry) — appends entry at head position, advances head with modulo wrap. If count < size, increments count. When full, oldest entry is silently overwritten. Thread-safe via mu.Lock()/defer mu.Unlock().
    - Lines() []LogEntry — returns a NEW allocated slice (snapshot) of all entries in chronological order (oldest first). When buffer not full (count < size), entries are at [0, count). When buffer is full (count == size), oldest entry is at head index — reconstruct order by copying entries[head:] then entries[:head]. Returns nil if count == 0. Thread-safe.
    - Len() int — returns count. Thread-safe.

    Test cases (input -> expected output):
    - newLogBuffer(5), write 3 entries -> Lines() returns 3 entries in order, Len() == 3
    - newLogBuffer(3), write 5 entries -> Lines() returns last 3 entries in chronological order, Len() == 3
    - newLogBuffer(1), write 3 entries -> Lines() returns only the last entry, Len() == 1
    - newLogBuffer(0) -> defaults to size 1000 (verify newLogBuffer returns non-nil with correct size)
    - newLogBuffer(-1) -> defaults to size 1000
    - Empty buffer -> Lines() returns nil, Len() == 0
    - Write with StreamStdout and StreamStderr -> Lines() preserves stream tags in order
    - Concurrent: 10 goroutines each write 50 entries while 1 goroutine continuously calls Lines() -> go test -race passes with no races detected

    Anti-patterns to avoid:
    - Do NOT use sync.RWMutex for the log buffer — writes and reads are roughly equal frequency, Mutex is simpler and faster at balanced ratios (per research)
    - Do NOT return the internal entries slice directly from Lines() — must allocate a new slice (snapshot) to prevent callers from mutating internal state
    - Do NOT use value receivers — sync.Mutex must not be copied after first use
  </behavior>
